    GitHub Copilot Prompting Workshop - Instructor Guide¶

    Workshop Overview¶

    Duration: 4 hours (flexible to 8 hours)
    Target Audience: Developers with mixed coding backgrounds
    Focus: Prompting skills, NOT coding skills
    Validation Method: Automated test suites
    Core Principle: Students never write code manually—only craft prompts

    Workshop Goals¶

    By the end of this workshop, participants will:

    1. Master all GitHub Copilot modes (Inline, Chat Ask/Plan/Agent, Beast Mode)
    2. Know when to use each mode for maximum effectiveness
    3. Craft effective prompts for different scenarios
    4. Configure custom instructions and agents
    5. Use .github directory for project-wide AI configuration
    6. Validate work through test-driven development
    7. Translate code across programming languages using AI

    Student Pre-requisites¶

    Required: - GitHub Copilot access (subscription or trial) - VS Code with GitHub Copilot extension - Python 3.12+ installed - Basic familiarity with terminal/command line

    Optional: - Node.js (for Exercise 15) - Git knowledge (helpful but not required)

    Send to students 2 days before:

If you encounter issues, please reach out before the workshop!

    Exercise-by-Exercise Teaching Guide¶

    Exercise 1: Hello Copilot - Inline Suggestions¶

    Learning Objective: Get comfortable with inline suggestion flow

    Teaching Points: - Copilot watches as you type - Tab accepts, Esc dismisses - Alt+[ and Alt+] cycle through suggestions - Comment-driven: write comment, then signature - Type hints help generate better suggestions

    Live Demo (5 min before students start): 

    # Show typing this slowly:
def multiply(x: float, y: float) -> float:
    # Pause here - Copilot suggests return x * y
    # Press Tab to accept

    Common Issues: - Copilot not suggesting: Check if enabled, try restarting VS Code - Wrong suggestions: Be more specific in function name/comment - Students accepting blindly: Remind them to READ before accepting

    Example Prompts to Share: 

    # Good:
def multiply(x: float, y: float) -> float:
    """Multiply two numbers."""
    # Copilot suggests correctly

# Better:
# Function to divide two numbers, handling division by zero
def divide(x: float, y: float) -> float:

    Expected Time: 10 min actual work, 5 min troubleshooting

    Exercise 2: Chat Ask - Quick Enhancements¶

    Learning Objective: When to switch from inline to Chat Ask

    Teaching Points: - Open Chat: Ctrl+Shift+I (Windows/Linux) or Cmd+Shift+I (Mac) - Chat Ask is for quick questions and additions - Reference “my calculator” to provide context - Iterate if first response isn’t perfect

    Live Demo (3 min): 

    [Open Chat]
"Add a modulo function that returns the remainder of division"

[Show Copilot's response, discuss if it's good]
[If not perfect]: "Make sure it handles modulo by zero"

    Common Issues: - Too vague: “add modulo” → Copilot unsure what to do - Wrong context: Copilot might suggest unrelated code - Not iterating: Accepting first response even if not ideal

    Example Prompts to Share: - ✅ “Add a modulo function that takes two numbers and returns remainder” - ✅ “How should I handle modulo by zero in my calculator?” - ❌ “add modulo” (too vague) - ❌ “make it better” (no direction)

    Expected Time: 10 min (may finish early)

    Exercise 3: Chat Iteration - Refinement¶

    Learning Objective: Iterative prompting for refinement

    Teaching Points: - First response is rarely perfect - Ask follow-up questions to refine - Be specific about what to improve - “Make error messages more helpful” vs “make it better”

    Live Demo (3 min): 

    [Chat]: "Add input validation to my calculator functions"
[Review response]
[Follow-up]: "The validation should provide specific error messages 
telling users what type was expected"

    Common Issues: - Giving up too early: Students accept mediocre first attempt - Too many changes at once: Ask for one improvement at a time - Not testing: Make changes without running code

    Discussion Point (after students complete): “How many prompts did it take to get it right? That’s normal! AI requires iteration.”

    Expected Time: 10 min

    Exercise 4: Chat Plan - Scientific Functions¶

    Learning Objective: Difference between Ask (quick) and Plan (complex)

    Teaching Points: - Plan mode is for architectural decisions - Get multiple options with pros/cons - Review plan BEFORE implementing - Can switch to Agent mode to implement plan

    Live Demo (5 min): 

    [Chat]: "Plan how to add scientific functions (sqrt, log, sin, cos, tan) 
to my calculator. Should they be in a separate module or integrated? 
Consider maintainability."

[Review Copilot's plan]
[Discuss options with students]
[Then]: "Implement option 2 from your plan"

    Common Issues: - Skipping planning: Going straight to implementation - Not reviewing options: Accepting first approach without thinking - Plan without implementation: Forgetting to actually implement

    Decision Matrix to Share: | Use Chat Ask | Use Chat Plan | |--------------|---------------| | Add one function | Add module/system | | Quick fix | Architectural change | | “How do I…” | “What’s the best way to…” |

    Expected Time: 15 min (planning takes time)

    Exercise 5: Documentation with Plan¶

    Learning Objective: Using Plan for non-code tasks

    Teaching Points: - Planning applies to documentation too - Copilot can generate README, docstrings, comments - Specify documentation style (Google, NumPy, etc.) - Documentation is code - treat it seriously

    Live Demo (3 min): 

    [Chat Plan]: "Plan a documentation strategy for my calculator project. 
What should be documented and how?"

[Review plan]
[Then]: "Implement the documentation plan. Add Google-style docstrings 
to all functions."

    Common Issues: - Incomplete docstrings: Missing Args or Returns sections - Generic docs: Not specific to the function - Forgetting README: Only documenting code, not project

    Quality Check Points: - All functions have docstrings - Docstrings include examples - README has usage instructions - Module has top-level docstring

    Expected Time: 12 min

    Exercise 6: Test-Driven Addition¶

    Learning Objective: Generating comprehensive test suites

    Teaching Points: - Tests are essential for validation - Copilot can generate entire test files - Specify: pytest, parametrized tests, edge cases - Run tests to verify they work!

    Live Demo (5 min): 

    [Chat]: "Generate a pytest test file (test_calculator.py) that tests all 
my calculator functions with normal cases, edge cases, and error scenarios. 
Use parametrized tests where appropriate."

[Show generated tests]
[Run pytest to verify]

    Common Issues: - Tests don’t run: Missing imports or syntax errors - Incomplete coverage: Only happy path tests - Not running tests: Generating but not executing - This takes longest: Students may need 15-20 min

    Troubleshooting Commands: 

    # If tests fail
pytest test_calculator.py -v  # Verbose output

# Check coverage
pytest --cov=calculator test_calculator.py

    Expected Time: 12-25 min (OFTEN TAKES LONGER - be flexible)

    BREAK (15 minutes after Exercise 6)¶

    Use this time to: - Check in with students who are behind - Answer questions - Show examples of good tests from faster students - Reset for Agent mode introduction

    Exercise 7: Agent Mode - Class Refactoring¶

    Learning Objective: Agent autonomy for complex, multi-step tasks

    Teaching Points: - Agent mode = autonomous multi-step execution - Provide complete requirements upfront - Agent will plan and execute on its own - Watch Agent’s narration to see its thinking - Can interrupt if going wrong direction

    Live Demo (5 min): 

    [Chat Agent]: "Refactor my calculator into object-oriented structure. 
Create a Calculator class with all operations as methods. Maintain all 
existing functionality and add a calculation history feature that tracks 
all operations performed."

[Watch Agent work]
[Point out Agent's steps]
[Show final result]

    When to Use Agent vs Ask/Plan: - Agent: Multi-file changes, complex refactoring, multiple related features - Ask: Quick additions, single function - Plan: Need to see options before deciding

    Common Issues: - Too vague prompt: Agent makes arbitrary decisions - Interrupting too early: Let Agent complete before judging - Not monitoring: Agent goes off track and student doesn’t notice

    Success Indicators: - Calculator is now a class - All methods work (tests pass) - History tracking exists and functions

    Expected Time: 15-20 min (Agent does the work, but students watch)

    Exercise 8: Agent Mode - Memory System¶

    Learning Objective: Agent handling multiple related features

    Teaching Points: - Agent can handle 4-5 related features in one prompt - Be explicit about all requirements - Specify error handling and edge cases - Agent will coordinate implementation across methods

    Example Prompts to Share: 

    Good: "Add history management with: show last N, clear history, export to 
file, recall last result. Each method should have error handling for empty 
history."

Bad: "add history stuff"

    Common Issues: - Missing requirements: Students forget to specify something, Agent omits it - Tests failing: Agent implementation doesn’t match test expectations - Incomplete features: Only implements 2 of 4 requested features

    Expected Time: 15 min

    Exercise 9: Agent Mode - Advanced CLI¶

    Learning Objective: Agent managing architectural changes

    Teaching Points: - CLI is complex: argparse, subcommands, config files - Let Agent handle the complexity - Specify user experience, not implementation - Agent will choose appropriate libraries

    Prompt Guidance: 

    "Implement an advanced CLI with:
- Interactive mode (continuous calculations)
- Single-calculation mode (one operation then exit)
- argparse for command-line arguments
- Config file support (config.json)
- Help text for all commands"

    Common Issues: - Too ambitious: Some students ask for too much - Config complexity: Config file support may be partial - Testing CLI: Hard to test interactively

    Discussion Point (after Ex 9): “What did you notice about Agent mode? When would you use it vs Chat Ask?”

    Expected Time: 15-20 min

    BREAK (5-10 minutes after Phase 3)¶

    Quick break before custom configurations

    Exercise 10: Custom Instructions Setup¶

    Learning Objective: Setting persistent project context

    Teaching Points: - .github/copilot-instructions.md = global context - Affects ALL Copilot interactions (inline, chat, agent) - Be specific: not “good style” but “Google docstrings with examples” - Include examples in instructions - Can update instructions as project grows

    Live Demo (5 min): 

    [Show creating .github/copilot-instructions.md]
[Add type hint requirements]
[Add docstring format]
[Add architecture principles]

[Then show Agent respecting instructions]:
"Add a plugin system following the instructions in 
.github/copilot-instructions.md"

    Instructions Template: 

    # Project Instructions

## Code Style
- Type hints: Python 3.12 syntax (use | for Union)
- Docstrings: Google style with examples
- Max line length: 100

## Architecture
- Class-based design
- Separate concerns
- Plugin extensibility

## Testing
- pytest with 90% coverage
- Parametrized tests

    Expected Time: 20 min (includes creating instructions file)

    Exercise 11: Instructions-Driven Enhancement¶

    Learning Objective: Using instructions for consistency

    Teaching Points: - Update instructions, then refactor code to match - Agent will follow instructions automatically - Good for maintaining consistency across team - Instructions as code: version control them

    Example Workflow: 1. Update instructions with stricter requirements 2. Use Agent: “Refactor entire codebase to match instructions” 3. Agent ensures consistency automatically

    Expected Time: 15 min

    Exercise 12-13: Custom Agents (If time permits)¶

    Learning Objective: Task-specific AI assistants

    Teaching Points: - Custom agents = specialized AI for specific tasks - Activate with @agent-name - Define role, approach, examples in YAML - Great for repeated task types (testing, docs, optimization)

    Live Demo (if time): 

    [Show test-agent.yml structure]
[Activate in Chat]: "@test-agent analyze my code for test gaps"
[Show specialized response]

    For Homework: If no time, assign Exercises 12-13 as practice

    Exercise 14-15: Beast Mode & JavaScript (Usually homework)¶

    If Time Permits (rare in 4-hour workshop): - Show Beast Mode as live demo (don’t have students do it) - Exercise 15 requires Node.js - assign as homework

    Demo Script: 

    [Beast Mode]:
"Add GUI (tkinter), database (SQLite), API (Flask), and unit conversions 
simultaneously. Plan integration first."

[Show Copilot handling multiple features]
[Point out complexity management]

    Common Issues & Troubleshooting¶

    Technical Issues¶

    Copilot Not Working: 1. Check if extension is enabled (bottom right status bar) 2. Verify GitHub Copilot subscription is active 3. Try reloading VS Code window 4. Check internet connection

    Tests Failing: 1. Read error message carefully 2. Use Chat: “Why is test_X failing?” 3. Check if imports are correct 4. Verify Python version (3.12+)

    Import Errors: 

    # Common fix
pip install pytest pytest-cov

# Or use requirements.txt
pip install -r requirements-dev.txt

    Pedagogical Issues¶

    Students Stuck: - Don’t give them code directly - Give them better prompts to try - Show example prompt for similar task - Encourage iteration: “Try asking Copilot to fix it”

    Students Going Too Fast: - Encourage them to help peers - Ask them to try alternative approaches - Give them bonus exercises (14-15) - Have them experiment with different prompts

    Students Falling Behind: - It’s okay to skip exercises - Focus on learning prompting, not completing all - Milestone 2 (Ex 6) is good stopping point - Provide completed code for next phase

    Prompt Quality Issues: - Show good vs bad prompt examples - Live demo of iterating a vague prompt - Reference good-vs-bad-prompts.md - Encourage specificity

    Workshop Variations¶

    For Experienced Developers (4 hours)¶

    • Move faster through Exercises 1-6 (1.5 hours)
    • Deep dive into Agent mode (Exercises 7-9)
    • Comprehensive custom configuration (Exercises 10-13)
    • Show Beast Mode as inspirational demo

    For Beginners (4 hours)¶

    • More time on basics (Exercises 1-6: 2 hours)
    • Simplified Agent exercises (7-9)
    • Skip custom agents (12-13)
    • Focus on prompting fundamentals

    Full Day Workshop (8 hours)¶

    • Complete all 15 exercises
    • More discussion and sharing time
    • Live debugging sessions
    • Student presentations of solutions
    • Deep dive into .github configurations

    Quick Reference Cards¶

    Copilot Modes Summary¶

    Mode When to Use Example
    Inline Writing new code, following patterns Function implementation
    Chat Ask Quick questions, small additions “Add modulo function”
    Chat Plan Architectural decisions, options “Plan how to add feature X”
    Chat Agent Multi-step, complex features “Refactor to OOP + add history”
    Beast Mode Multiple features simultaneously Advanced users only

    Troubleshooting Checklist¶

    • Copilot extension enabled?
    • Python 3.12+ installed?
    • pytest installed?
    • Tests running: pytest tests/test_exN.py -v
    • Prompts are specific and contextual?
    • Student reading errors before asking for help?

    Remember: This workshop is about prompting mastery, not coding skill. Success = students can craft effective prompts for any Copilot mode. Guide them to discover better prompting through iteration and experimentation!
